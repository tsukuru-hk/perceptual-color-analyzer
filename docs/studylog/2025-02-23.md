# 今日の会話から得た学び（設計・運用・判断）

今日の会話全体を振り返り、初学者として学びになったポイントをまとめた。

## 設計・構成の考え方

- **「複雑にしすぎていないか」を定期的に問う**: 機能の数や人数に比べて、docs やフォルダが多すぎると追いづらくなる。まずは「本当に必要か」「今の規模で足りているか」を確認する。
- **レイヤー型 vs フィーチャーベース**: レイヤー型は「依存の向きが分かりやすい」が、機能が増えると 1 機能の変更が複数フォルダに散る。フィーチャーベースは「1 機能が 1 フォルダにまとまる」が、共有部分の置き場所の判断が増える。このプロジェクトでは「ドメイン・ユースケースは層のまま、UI だけ機能単位（features/）にまとめる」ハイブリッドを採用した。
- **Container / Presentation 分離**: 状態とユースケース呼び出しは Container、見た目だけは Presentation。分離をやめるか残すかは「追いやすさ」と「役割の明確さ」のどちらを優先するかで決める。小規模でも「型」として残しておくと、あとで機能を増やすときのテンプレートになる。
- **UseCase と composable の違い**: UseCase は「アプリのシナリオ」（Vue に依存しない、Result で返す）。composable は「画面まわりの状態・ロジックの再利用」（ref や DOM を扱う）。ユースケースを呼ぶのは Container。composable はユースケースを呼ばない。役割を分けておくと、テストや差し替えがしやすくなる。

## ドキュメントと運用

- **「何を採用するか」を一つずつ決める**: 複数の方針が並んだときは、一つずつ「問題点・変更すべき理由・変更すべきでない理由」を並べてから判断すると、初学者でも選びやすい。
- **細かい変更は docs に書かなくてよい**: 存在しないファイルへのリンクの削除など、細かい修正は必ずしも worklog や README に追記しなくてよい。重要な判断や「なぜこうしたか」だけ残す。
- **worklog のファイル名で内容が分かるようにする**: `2025-02-23.md` より `2025-02-23 実装まとめと色パネル改善.md` のように、ファイル名からざっくり何をしたか分かると、あとで探しやすい。
- **実装や空箱は依頼があるまで作らない**: 「共通コンポーネント・composable の構造」を決めても、空のファイルやスタブを勝手に増やさない。ディレクトリだけ用意し、中身は依頼時に作る。そうすると「何が方針で何が実装済みか」が混ざらない。

## 命名と慣習

- **features と image-analysis の二段階**: `features/` は「機能を入れる入れ物」、`image-analysis/` は「画像解析」という 1 つの機能の名前。機能が増えたら `features/grayscale-map/` のように同じ階層に足していく。
- **Vue のブロック順**: template → script → style に揃えると、どのファイルも同じ順で読める。プロジェクトのルールとして code-rules に書いておく。

## UI・スタイル

- **テーマとコンポーネントの色**: アプリ全体が暗いテーマ（`:root` で白文字）なのに、一部だけ薄い背景（#f5f5f5）にすると、文字が継承で白のままになり見えなくなる。パネルも暗いトーンに揃えるか、パネル内で文字色を明示する。
- **scoped で色を明示する**: コンポーネントの見た目は、できるだけそのコンポーネントの `<style scoped>` で完結させると、グローバルスタイルの変更に振り回されにくい。

## その他

- **AGENTS.md をルートに置く理由**: Cursor など多くのツールが「ルートの AGENTS.md」を AI 用の説明として読む慣習がある。docs にまとめたい場合は移せるが、ツールのデフォルトに合わせるならルートのままが無難。
- **studylog の役割**: 本番コードや worklog には書かない「調べもの」「試したコード」「学習メモ」を置く。運用で必須にしなくてよい。

# 2025-02-23 実行の観察とブレークポイント

## 何をしたか

- 実行の観察・エラー検知のやり方を依頼→最初は実装まで入れたが「教えるだけ、studylog だけ」と指摘されロールバック。
- 壁打ちで観察の一般論（4 方法）を聞き、まずブレークポイントを覚えるとアドバイスされた。
- このプロジェクトでの手順（Sources → ファイル選択 → 行番号左クリック → 操作で止める → Scope/Call Stack を見る → 再開）を試した。
- `?t=` 付きファイルが複数ある問題、「止まる」の意味・見え方を質問して解消。成功/失敗の観測の仕方とブレークポイントの用途を整理した。

## 何を学んだか

- **観察の方法**: ログ / ブレークポイント / コールスタック / Network。まずブレークポイント。
- **手順**: Sources → 該当ファイル → 行番号左でブレーク → その行を通る操作をすると止まる。青く光る・矢印・Paused in debugger・Scope/Call Stack が出る。▶ で再開。
- **`?t=`**: Vite のキャッシュ用。どれでもよい。止まらなければ数字が最大のものを選ぶ。
- **成功/失敗**: 画面＝OKLCH 表示 or 赤エラー。デバッガー＝`success`/`failure` の行で止めて value や error を見る。
- **用途**: 特定箇所がうまくいっているか・どういう状態か確かめるときに使う。

## 何に詰まったか

- 同じファイルが `?t=` で複数→どれを選ぶか。→ どれでもよい / 止まらなければ最新の `?t=`。
- 「止まる」が観測できない。→ 青ハイライト・矢印・Paused・Call Stack/Scope の変化で分かる。実際にクリックして確認した。
