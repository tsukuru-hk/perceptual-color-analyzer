# 2025-02-23 実行の観察とブレークポイント

## 何をしたか

- 実行の観察・エラー検知のやり方を依頼→最初は実装まで入れたが「教えるだけ、studylog だけ」と指摘されロールバック。
- 壁打ちで観察の一般論（4 方法）を聞き、まずブレークポイントを覚えるとアドバイスされた。
- このプロジェクトでの手順（Sources → ファイル選択 → 行番号左クリック → 操作で止める → Scope/Call Stack を見る → 再開）を試した。
- `?t=` 付きファイルが複数ある問題、「止まる」の意味・見え方を質問して解消。成功/失敗の観測の仕方とブレークポイントの用途を整理した。

## 何を学んだか

- **観察の方法**: ログ / ブレークポイント / コールスタック / Network。まずブレークポイント。
- **手順**: Sources → 該当ファイル → 行番号左でブレーク → その行を通る操作をすると止まる。青く光る・矢印・Paused in debugger・Scope/Call Stack が出る。▶ で再開。
- **`?t=`**: Vite のキャッシュ用。どれでもよい。止まらなければ数字が最大のものを選ぶ。
- **成功/失敗**: 画面＝OKLCH 表示 or 赤エラー。デバッガー＝`success`/`failure` の行で止めて value や error を見る。
- **用途**: 特定箇所がうまくいっているか・どういう状態か確かめるときに使う。

## 何に詰まったか

- 同じファイルが `?t=` で複数→どれを選ぶか。→ どれでもよい / 止まらなければ最新の `?t=`。
- 「止まる」が観測できない。→ 青ハイライト・矢印・Paused・Call Stack/Scope の変化で分かる。実際にクリックして確認した。
