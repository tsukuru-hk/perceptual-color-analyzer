# src/ 構造の比較：レイヤー型 vs フィーチャーベース型

現状の **レイヤー型** と、設計思想で触れた **フィーチャーベース型** を比較し、このリポジトリにどちら（またはハイブリッド）が向くかを整理する。

---

## 1. 二つの形のイメージ

### A. レイヤー型（現状）

```
src/
├── core/                    # 共通基盤（Result, 例外）
├── domain/                  # ドメイン層（OKLCH など）
├── application/useCase/    # アプリケーション層（ユースケース）
├── infrastructure/          # インフラ層（画像読み込み、ピクセル取得）
├── types/                   # 型定義
├── components/              # Vue コンポーネント（横断）
├── App.vue
└── main.ts
```

- **軸**: 「役割（層）」で分割。どこに何があるかが層で決まる。
- **依存の向き**: 画面 → Application → Domain ← Infrastructure と一方向。

### B. フィーチャーベース型（設計思想で言及）

```
src/
├── core/                    # 共通基盤（または shared/）
├── features/
│   ├── image-upload/        # 機能単位
│   │   ├── container/       # 状態・ロジック
│   │   ├── components/      # 表示
│   │   ├── composables/
│   │   ├── utils/
│   │   └── index.ts
│   └── pixel-picker/
│       ├── container/
│       ├── components/
│       └── ...
├── App.vue
└── main.ts
```

- **軸**: 「機能（フィーチャー）」で分割。一つの機能に container / components / composables / utils をまとめる。
- **共有**: ドメイン・ユースケース・インフラは `core/` や `shared/` に置くか、各 feature から参照。

---

## 2. 比較表

| 観点 | レイヤー型（現状） | フィーチャーベース型 |
|------|-------------------|----------------------|
| **依存の把握** | 層ごとに依存方向が明確。DDD/クリーンアーキテクチャに沿う。 | 機能内は閉じるが、共有レイヤーとの依存は feature 数が増えると追いづらくなりがち。 |
| **変更の局所性** | 一つの機能を直すのに、domain / application / infrastructure / components を横断して触ることが多い。 | 一つの機能はだいたい一つの feature 配下にまとまる。変更が feature 内に収まりやすい。 |
| **「どこに書くか」の判断** | 層のルールが分かれば決めやすい（「これはインフラ」「これはユースケース」）。 | 新機能は「新 feature を作る」で済む。既存機能の共通化は「shared に出すか、どちらの feature に置くか」の判断が発生。 |
| **規模が小さいとき** | ファイル数が少なく、層がそのままナビゲーションになる。シンプル。 | feature を細かく分けるとディレクトリだけ多くなり、小規模だと過剰になりがち。 |
| **規模が大きいとき** | 層ごとにファイルが増える。「画像まわり」が application / infrastructure / components に散る。 | 機能ごとにまとまるので、「画像アップロード」は一つの feature を見ればよい。 |
| **AI・人間のナビ** | 「ユースケースは application/useCase」「インフラは infrastructure」と一言で案内できる。 | 「〇〇機能は features/〇〇」と案内できる。参照する単位が「機能」になる。 |

---

## 3. このリポジトリの文脈で考える

- **現状**: フロントのみ。バックエンドなし。domain / application / infrastructure はすでにあり、依存の向きも README で明文化されている。
- **ロードマップ**: 彩度グレースケール、3D ガマット・グラフなど、**機能が縦に増えていく**想定。

| 選択肢 | 向いている場合 |
|--------|----------------|
| **レイヤー型を維持** | ドメイン・ユースケースの「層」を崩したくない。機能数がまだ少なく、横断的な変更も許容できる。 |
| **フィーチャーベースに寄せる** | 「画像アップロード」「ピクセルピッカー」「彩度可視化」「3Dガマット」など、機能ごとに開発・参照したい。 |
| **ハイブリッド** | **ドメイン・アプリケーション・インフラはレイヤー型のまま**置き、**フロントの「画面・UI のまとまり」だけ** features/ で分ける。 |

---

## 4. ハイブリッド案（推奨のイメージ）

「ビジネスロジックは層を保ち、UI のまとまりは機能単位でまとめる」形。

```
src/
├── core/                    # そのまま
├── domain/                  # そのまま
├── application/useCase/     # そのまま
├── infrastructure/          # そのまま
├── types/                   # そのまま
├── features/                # フロントのまとまり
│   ├── image-analysis/      # 画像アップロード + キャンバス + ピクセルOKLCH
│   │   ├── ImageAnalysisContainer.vue   # 状態・ユースケース呼び出し
│   │   ├── ImageCanvas.vue
│   │   ├── PixelOklchResult.vue
│   │   └── index.ts
│   └── (今後) chroma-visualization/
│       └── ...
├── App.vue                  # features を組み合わせるだけ
└── main.ts
```

- **メリット**: 依存の向き（画面 → Application → Domain ← Infrastructure）を維持したまま、**どの画面がどのユースケースを使うか**が feature 配下で見える。機能追加時は「新 feature を足す」か「既存 feature に足す」で判断できる。
- **デメリット**: 「レイヤー」と「フィーチャー」の二軸が残るので、ルールを docs に書いておく必要がある。

---

## 5. まとめ

| 選び方 | 目安 |
|--------|------|
| **いまのレイヤー型のまま** | 機能が 2〜3 個程度で、層の明確さを最優先するならそのままでよい。 |
| **ハイブリッドに進める** | 彩度可視化・3D ガマットなど機能が増えそうなら、**フロントだけ** features/ を導入するのがおすすめ。domain/application/infrastructure は触らない。 |
| **全面フィーチャーベース** | ユースケースやインフラも feature に寄せる場合は、共有用の core/ を用意し、「依存の向き」を docs で明文化する必要がある。現状規模だと移行コストの割にメリットは小さめ。 |

**結論**: 現状の「層」はそのままにして、**フロントのコンポーネント配置だけ**「`components/` 横並び」から「`features/{名前}/` でまとめる」に変える **ハイブリッド** が、このリポジトリには扱いやすい。
